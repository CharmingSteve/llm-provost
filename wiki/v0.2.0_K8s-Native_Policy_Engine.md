# v0.2.0: K8s-Native Policy Engine

## Architecture Overview
Moving from docker-compose to a K8s Deployment with an OpenResty container and a ConfigMap-driven policy.

These are the exact policies your OpenResty proxy will enforce by parsing the JSON payloads, and what your Prometheus/Grafana stack will monitor from the logs:

### 1. Static Limits (The Stateless Checks)
These are evaluated instantly by looking at the single JSON request payload:
*   **Max Quantity per order:** A hard ceiling on how much can be moved in one API call (e.g., blocking `qty > 100`).
*   **Restricted Symbols / Assets:** A "No-Fly List" blocking specific tickers, meme stocks, or unauthorized digital assets.

### 2. Stateful / Cumulative Limits (The "Hard" Part)
These require OpenResty's `lua_shared_dict` to track the JSON data across multiple requests in real-time, without needing an external database:
*   **Daily Notional Value:** *"Don’t let the AI trade more than $50k total today."* The Lua layer parses the JSON, calculates the value, and adds it to a running daily total in memory.
*   **Max Open Positions:** *"Don’t let the AI hold more than 5 assets at once."* 

### 3. Heuristic / AI-Specific Rules (The Hallucination Guard)
These rules are specifically designed to catch the unique failure modes of Large Language Models:
*   **Duplicate Detection:** *"If the LLM sends the exact same JSON order 5 times in 1 minute, block it."* This is the circuit breaker that prevents an AI from getting stuck in an "Infinite Loop" hallucination and draining an account in seconds.

```json
      "ticker_blacklist": ["GME", "AMC", "BBBY"]

      "max_open_positions": 5

        "window_seconds": 60
      }
    }
  }

```

## The 'Atomic Reload' Workflow
Using ArgoCD PostSync hooks to trigger a POST to `/_internal/reload_policy` on the proxy pods.

## Lua Logic (OpenResty)
- `init_worker_by_lua`: Load policy from `/etc/provost/policies.json` on startup.
- `/_internal/reload_policy`: Internal endpoint to reload the ConfigMap file into `lua_shared_dict` memory.
- `access_by_lua`: Real-time enforcement of:
    1. **Static Limits:** Quantity checks and ticker blacklisting.
    2. **Stateful/Cumulative Limits:** Daily notional tracking and open position counts via `lua_shared_dict`.
    3. **Heuristic/AI-Specific Rules:** Duplicate detection to prevent infinite loop hallucinations (e.g., blocking the same order if repeated > 5 times in 60s).
